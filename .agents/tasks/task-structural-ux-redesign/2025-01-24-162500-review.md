# Structural UX redesign of the review workspace

The workspace layout shifts from a flat chat-centric view to a 3-tab information architecture (Overview / Findings / Technical Activity). The landing page now creates a session directly via API and navigates to it with auto-start, removing the intermediate sessions list redirect. New components decompose the review results into a summary card, top findings, and next-steps panel. The findings tab replaces the old 2-column artifact grid with a vertical list of severity-badged `FindingCard` components that support status changes, clipboard copy, and an expandable detail overlay.

**Watch for:** The status dropdown in `FindingCard` lacks a click-outside handler, so it stays open if the user clicks elsewhere on the page (confirmed). The `parseLocation` regex in `FindingCard` will false-positive on ordinary prose that mentions file-like strings, potentially extracting misleading "locations" (likely). The landing page duplicates both the form and the error display JSX rather than extracting a shared component (confirmed, minor).

## High-level view

The tab architecture replaces the old `debate` / `artifacts` / `results` triple with `overview` / `findings` / `technical`. The Overview tab acts as a state machine: progress spinner while active, full report when consensus exists, fallback with partial findings otherwise.

The landing-to-review flow makes a POST to `/api/sessions` directly from the home page — URL input to live analysis in one click. The `?start=1` query parameter triggers auto-start inside `WorkspaceLayout` via a `useEffect` that fires once, then rewrites the URL. This couples page navigation state to a side-effect; if the session row isn't committed before the redirect lands, SWR could briefly 404 (possible race, mitigated by the `await` in `handleAnalyze`).

The `FindingCard` component parses severity from artifact type (statically — ignoring any backend severity data), extracts a file location via regex, builds a copyable "task" string, and supports optimistic status updates with rollback on failure. The status dropdown has no click-outside dismissal, and the regex extraction is overly broad. The `TechnicalActivityPanel` absorbs the agent arena, debate chat, budget bar, and round-indicator dots that previously lived inline in `WorkspaceLayout`.

The hero and bottom CTA forms on the landing page are copy-pasted (~60 lines each) with slightly different error-message styling. Divergence between these copies is inevitable.

<summary>Issues (5)</summary>

1. **FindingCard dropdown stays open on outside click** — The status dropdown has no `useEffect` with a click-outside listener or `onBlur` handler, so clicking anywhere outside the dropdown leaves it visible. Add a click-outside handler or use a headless UI popover.
2. **parseLocation false positives** — The regex matches file-like substrings in prose (e.g., "next.js", "v2.0", "e.g."). Gate extraction on stricter heuristics (require a directory separator or known extension list).
3. **Duplicated form + error markup on landing page** — The hero form and the bottom CTA form are copy-pasted with minor styling differences. Extract a `RepoForm` component to keep them in sync.
4. **Optimistic status has no timeout** — `handleStatusChange` in `FindingCard` awaits the fetch indefinitely. If the server hangs, `isUpdating` stays true forever and the button remains disabled. Add an `AbortController` with a timeout.
5. **Auto-start effect depends on render timing** — The `?start=1` auto-start relies on `isEmptyState` being true on the first render. If session creation succeeds but the redirect lands before the server commits the row, the page could briefly 404 from SWR. Unlikely due to the `await`, but a retry or error boundary would make it resilient.

## FindingCard dropdown and interaction gaps

The status dropdown opens via `setShowStatusDropdown(true)` on button click, but the only close paths are: (a) selecting a status option, or (b) clicking the toggle button again. No document-level click listener, no `onBlur`, no overlay backdrop.

```tsx
// Only close path besides re-click:
const handleStatusChange = async (status: ArtifactStatus) => {
  setShowStatusDropdown(false); // <-- only here
  ...
};
```

The card itself has `onClick={() => setExpanded(true)}` on the outer div, with the actions row calling `e.stopPropagation()`. Clicking the card body while a dropdown is open will open the detail modal but leave `showStatusDropdown` as `true`. On re-render after detail close, the dropdown will flash open briefly.

## Location extraction heuristic

`parseLocation` uses a broad regex:

```
/(?:^|\s|`)((?:\.{0,2}\/)?(?:[\w@.-]+\/)*[\w@.-]+\.\w{1,10})(?:\s|`|$|,|:)/m
```

This matches `react.19`, `v2.0`, `e.g.`, or `next.js` in ordinary prose. Since artifact content is freeform text written by AI agents, false positives are likely. The consequence is a misleading file-location badge on the card — cosmetic, but erodes trust in the UI.

## Severity mapping ignores backend data

`typeSeverityMap` maps artifact type to severity statically: all `risk` = "high", all `decision` = "medium". If the backend sends a low-severity risk (the `ArtifactState` type likely carries severity from the consensus engine), the UI still badges it "high". The mapping should eventually yield to data-driven severity when the field is available.

## Optimistic status has no timeout or abort

The `fetch` in `handleStatusChange` has no `AbortController` and no `setTimeout` fallback. If the server takes 30+ seconds, the card shows the optimistic state with a permanently disabled button. A 10-second abort with rollback would prevent the UI from getting stuck.
