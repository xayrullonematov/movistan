# UI/UX density reduction and tool-call visibility

This change implements the P2DESIGN.md playbook: a headline recommendation on the Results screen, a new ToolCallTrace component rendering live and completed tool-call data, collapsed-by-default debate messages, capped Key Decisions / Risk Register lists, a default "accepted" filter on artifacts, replacement of placeholder icons with real SVG paths, and deletion of three dead components. The approach is additive to existing patterns — accessibility conventions from AgentArena are adopted, design tokens are reused, and the tab structure is untouched.

**Watch for:** Debate messages start collapsed with no mechanism to expand all at once, meaning long sessions require N individual clicks to read the full thread (confirmed). The ToolCallTrace `parseStageProgress` silently drops events where `groundedByRepo` is falsy, which may hide agents that ran tools but weren't grounded by a repo (likely). No component tests were added for the new interactive behaviors (confirmed).

## High-level view

Debate messages now render as a one-line summary by default, with full structured content behind an expand click. The implementation wires `aria-expanded`, keyboard handlers, and focus-visible rings matching AgentArena's conventions, but provides no "expand all" affordance — a UX gap for power users who want to read the full thread. The summary derivation falls back to `content.slice(0, 100)` which will display raw JSON fragments for structured messages lacking a `summary` field.

ToolCallTrace is a brand-new collapsible widget placed at the top of the debate scroll area. It parses `stage-progress` events from the event stream for tool-call metadata, filtering to only those flagged `groundedByRepo: true`. The filtering choice means an agent that executed tool calls without repository grounding will be invisible in the trace, with no indication anything was dropped.

The ResultsDashboard headline derives confidence from `topDecision.confidence * 100` — if `confidence` is already stored as a percentage rather than a 0-1 fraction, the display would be nonsensical. Both Key Decisions and Risk Register cap at 5 items with accessible "show all" expanders.

No tests were added for any of the new interactive behaviors, which directly contradicts the P2DESIGN acceptance criteria requiring "new tests added for any new collapse/filter logic."

## DebateMessage collapse and the "expand all" gap

In a 4-agent round with critique + revision stages, the debate view can easily contain 12-16 messages. Requiring individual clicks to read any of them punishes anyone who actually wants to review the reasoning. A single "Expand all / Collapse all" toggle at the top of the DebateChat scroll area would preserve the progressive-disclosure principle while respecting power users — a shared state lifted to DebateChat and passed down as a prop.

The summary derivation has a fragility concern. When `parsed` doesn't have a `summary` field, it falls back to `content.slice(0, 100)`. Since `content` is a raw JSON string (the `PersistedEvent.content` field is typed as `string` containing serialized JSON), the collapsed preview for structured agent outputs will show something like `{"position":"We recommend...","objections":[{"point":`. The fix is to try extracting the first meaningful text field from the parsed structure (e.g., `parsed.position`, `parsed.recommendation`, `parsed.assessment`) before falling through to the raw slice.

## ToolCallTrace event filtering

```typescript
function parseStageProgress(event: PersistedEvent): StageProgressData | null {
  try {
    const data = JSON.parse(event.content);
    if (!data.groundedByRepo) return null;  // ← drops all non-grounded events
    ...
```

The P2DESIGN motivation is "surface the tool-call grounding visibly" — so filtering to grounded-only is intentional. But if an agent ran 15 tool calls and wasn't flagged as grounded (perhaps a configuration or data bug), it disappears entirely. The UI gives no hint that events were filtered. A "3 events hidden (not repo-grounded)" note would make the filtering transparent and help debug data-layer issues.

The `agentId` fallback (`data.agentId || event.agentId || "senior-engineer"`) silently defaults to senior-engineer when both fields are missing, rendering a malformed event as "Engineer" with the blue dot — misleading attribution.

## ResultsDashboard confidence scale

The headline computes `Math.round(topDecision.confidence * 100)`. If `confidence` is already stored as a percentage (0–100) rather than a 0–1 fraction, the headline would display values like "confidence 7200%." The ConsensusOutput type doesn't constrain the range. A guard (`confidence > 1 ? confidence : confidence * 100`) or a domain-level assertion would prevent this from surfacing as a confusing display bug (possible).

## Missing tests

The P2DESIGN acceptance criteria explicitly require: "No regressions in existing component tests; new tests added for any new collapse/filter logic." This diff adds zero tests. Untested behaviors:

- DebateMessage collapse/expand state toggle and keyboard accessibility
- ToolCallTrace event parsing, filtering, and fallback logic
- ResultsDashboard capping and "show all" toggling
- WorkspaceLayout default filter value
